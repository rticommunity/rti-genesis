How It Works

You don't configure your agent network — you grow it

Most frameworks require you to define your agent topology upfront: which agents exist, who can talk to whom, and in what order. In GENESIS, you write a new agent, start it, and the rest of the network discovers it automatically. The system adapts.

Under the hood: GENESIS replaces brittle HTTP wiring and manual service registries with RTI Connext DDS — the same peer-to-peer middleware trusted in surgical robots, flight control systems, and autonomous vehicles. Every agent gets industrial-grade reliability for free.
🔍

Zero-Config Discovery

Agents publish capabilities to durable DDS topics at startup. Any component joining the network instantly reads the full catalog — no configuration files, no IP lists, no restart required.

🤖

Core Innovation

The Agent-as-Tool Pattern

In most multi-agent frameworks, orchestration is hand-written: a coordinator decides which agent to call, formats the request, and stitches results back together. This logic breaks every time the network changes.

GENESIS eliminates the orchestration layer. Every agent discovered on the DDS network is automatically converted to a tool schema and injected into the primary LLM's call alongside local tools and service functions. The LLM decides what to call — in natural language. Routing happens invisibly over DDS RPC.

Result: Adding a new specialist agent to a running network requires zero changes to any existing agent. It appears as a new tool on the next LLM call automatically — the system's intelligence grows with every agent you start.
Python 
# The primary agent — no wiring required
class PrimaryAgent(OpenAIGenesisAgent):
    def __init__(self):
        super().__init__(
            model_name="gpt-4o",
            agent_name="PrimaryAgent",
            # auto-discovers all agents on network:
            enable_agent_communication=True
        )

# A specialist on any machine — auto-discovered
class WeatherAgent(OpenAIGenesisAgent):
    @genesis_tool(description="Get weather for a city")
    async def get_weather(self, city: str) -> dict:

Developer Experience

Zero boilerplate. Pure Python.

Two decorators cover the entire integration surface. GENESIS infers the JSON schema from your Python type hints and docstrings, advertises the capability over DDS, and makes it callable from any other agent on the network — all at import time.

@genesis_function exposes a method on a service as a distributed RPC call.
@genesis_tool adds an agent method to the LLM's tool list directly.

Adding a new LLM provider requires implementing 7 abstract methods (~150 lines). Currently supported: OpenAI, Anthropic (Claude), and Nemotron (via Ollama) for fully local, air-gapped inference.

Python 
# Service: expose a function over DDS
from genesis_lib import EnhancedServiceBase
from genesis_lib.decorators import genesis_function

class DataService(EnhancedServiceBase):
    @genesis_function()
    async def query(self, sql: str) -> dict:
        """Run a SQL query and return results."""
        ...

# Agent: add a tool to the LLM's toolset
from genesis_lib.decorators import genesis_tool

class AnalysisAgent(OpenAIGenesisAgent):
    @genesis_tool(description="Summarize a dataset")
    async def summarize(self, data: str) -> dict:
        """Return key statistics for the given data."""

Architecture

Three layers. One DDS fabric.

GENESIS separates transport, observability, and provider logic into discrete layers — so swapping an LLM provider never touches routing or monitoring code.

FAQ

Frequently asked questions

Can't I just use MCP?

MCP (Model Context Protocol) solves a different problem. It standardizes how a single LLM accesses tools and data — think file systems, databases, web search. The LLM calls a tool, gets a result, moves on. MCP servers are typically stateless and synchronous.

GENESIS solves how agents communicate with other agents across a distributed network. A GENESIS agent is a full stateful actor with its own LLM, memory, and tools. It discovers peer agents automatically, delegates goals to them, and coordinates across machines in real time — with guaranteed delivery and sub-millisecond latency that HTTP-based protocols cannot match.

The two are complementary: any GENESIS agent can expose itself as an MCP server via agent.enable_mcp(), making it callable from Claude Code, Cursor, or any MCP client. MCP handles the tool layer; GENESIS handles the agent network layer.

What about A2A?

A2A (Agent2Agent Protocol) is an HTTP-based standard for agents built on different frameworks to delegate tasks to each other over the internet. It is a useful interoperability standard for cloud-native, cross-organization agent communication.

GENESIS operates at a different level. Rather than a communication protocol layered over HTTPS, it uses RTI Connext DDS — a peer-to-peer, broker-free transport with configurable QoS, sub-millisecond latency, and guaranteed delivery. Discovery is automatic and zero-configuration; no Agent Card lookup, no known URL required. Agents find each other the moment they start.

A2A is well suited for loosely coupled cloud agents across organizational boundaries. GENESIS is built for tightly integrated, real-time, distributed systems — robotics, defense, industrial control — where HTTP latency and cloud dependency are not acceptable. The two serve different environments and are not in direct competition.

Other frameworks also connect agents over HTTP — what's different here?

HTTP is convenient but carries structural limitations for distributed agent systems: it requires a broker or service registry, has no built-in delivery guarantees, and relies on timeout-based failure detection. If a component goes offline mid-request, the caller has no reliable way to know.

GENESIS is built on RTI Connext DDS — the OMG-standard middleware deployed in FDA-cleared medical devices, NATO programs, and autonomous vehicle platforms. Every connection in a GENESIS network gets:

  • Configurable delivery guarantees — best-effort or fully reliable, per topic
  • Liveliness heartbeats — dead agents are detected automatically, not by timeout
  • TRANSIENT_LOCAL durability — a late-joining agent instantly receives the full history of what it missed
  • Sub-millisecond latency — via shared memory on the same host or UDP across a LAN
  • No broker — true peer-to-peer; no single point of failure

For enterprise web applications, HTTP is fine. For systems where reliability, latency, and resilience are non-negotiable, DDS is the right foundation.

Do I need to reconfigure my system when I add a new agent?

No — that's the core design principle of GENESIS. Write a new agent, start it, and every other agent on the network discovers it automatically via DDS. The primary agent's LLM sees the new capability on its next call and can start using it immediately, without any changes to existing agents or configuration files.

This is fundamentally different from frameworks where you define your agent topology upfront in a config file or YAML. In GENESIS, the topology is emergent — the network reorganizes itself as agents join and leave. Your system's intelligence grows with every agent you start.

What LLMs does GENESIS support?

GENESIS currently ships with three providers:

  • OpenAI — GPT-4o and any OpenAI chat completions model
  • Anthropic — Claude models via the Anthropic API
  • Nemotron (via Ollama) — fully local inference, no API key, no internet connection required

Adding a new provider requires implementing 7 abstract methods — roughly 150 lines of Python. The provider layer is cleanly separated from the transport and monitoring layers, so swapping or extending LLM support never touches your agent's communication or observability code.

Do I need an internet connection or cloud to run GENESIS?

No. GENESIS runs entirely on-premise or on disconnected edge hardware. RTI Connext DDS is a peer-to-peer protocol — it requires no cloud service, no broker, and no internet connectivity. Agents discover each other automatically on a local network.

For LLM inference, Ollama provides fully local model execution with no API key or external calls. Combined with DDS transport, a GENESIS deployment can operate in a fully air-gapped environment — which is a hard requirement in defense, classified, and certain industrial deployments.

Is GENESIS only for defense and aerospace?

No — GENESIS originated from a U.S. SBIR contract, which is why defense and aerospace are a natural fit. But the underlying capabilities are valuable wherever you need distributed, reliable, real-time agent communication:

  • Robotics and autonomous systems — drone fleets, industrial robots, autonomous vehicles
  • Industrial control and edge AI — process automation, smart manufacturing, power grid monitoring
  • Healthcare — clinical workflow agents that connect siloed EMR, imaging, and insurance systems
  • Simulation and digital twins — LLM agents interacting with DDS-based simulation environments
  • Enterprise AI pipelines — any organization that needs agents running across multiple machines with guaranteed communication

If your use case requires agents on separate machines to find each other automatically, communicate reliably, and scale without reconfiguration — GENESIS is built for it.
